# Nijo App Scaffold デモプロジェクト

## 番号

- 000～: 基本的な説明
- 100～: 実用的な例
- 200～: エッジケース。主に動作確認目的

## デモ新規作成時

1. `nijo new <プロジェクト名>` でプロジェクトを作成
2. 不要なファイルを削除
   - README.md
3. client の package.json の編集
   - nameを変更
   - dependencies, devDependencies を削除。モノレポのルートの設定を援用するので
4. [VSCode設定ファイル](../nijo.code-workspace) の `"id": "demoFolder"` の箇所にフォルダ名を追加。VSCode > F1 > Run Task からのデバッグコマンド等で新規追加プロジェクトを選択できるようにする
5. [VSCode設定ファイル](../nijo.code-workspace) の `"folders"` に新規追加プロジェクトのフォルダを追加（任意）
6. [モノレポのルートのpackage.json](../package.json) に新規追加プロジェクトのclientフォルダを追加
7. .NET の整備 ... 後述※1のコマンドを参考にして、以下のことを行う。
   1. エクスプローラーで新たに作成された `MyApp.sln` のファイル名を変更
   2. そのソリューションに含まれる各プロジェクトの名前を変更
   3. [.NET のソリューション](../nijo.sln) に追加
8. 一度clientフォルダ直下で `npm run dev` を行い node_modules フォルダを生成しておく。デバッグ開始時に未インストールと判定して npm ci しに行こうとするため

※1: .NET の整備コマンド

```sh
cd demo/999_NEW_DEMO

# 1. ファイルのリネーム
mv MyApp.sln Demo999.sln
mv Core/MyApp.csproj                             Core/Demo999.csproj
mv Core.AutoGenerated/MyApp.AutoGenerated.csproj Core.AutoGenerated/Demo999.AutoGenerated.csproj
mv WebApi/MyApp.WebApi.csproj                    WebApi/Demo999.WebApi.csproj
mv UnitTest/MyApp.UnitTest.csproj                UnitTest/Demo999.UnitTest.csproj
mv Management/MyApp.Management.csproj            Management/Demo999.Management.csproj

# 2. .csproj ファイル内の参照パスを書き換え (sedを使用)
sed -i 's/MyApp.AutoGenerated.csproj/Demo999.AutoGenerated.csproj/g' Core/Demo999.csproj
sed -i 's/MyApp.csproj/Demo999.csproj/g'                             WebApi/Demo999.WebApi.csproj
sed -i 's/MyApp.csproj/Demo999.csproj/g'                             UnitTest/Demo999.UnitTest.csproj
sed -i 's/MyApp.csproj/Demo999.csproj/g'                             Management/Demo999.Management.csproj

# 3. Demo999.sln から古い参照を削除し、新しいプロジェクトを追加
# (removeコマンドはファイルが存在しなくてもslnからエントリを削除してくれます)
dotnet sln Demo999.sln remove \
  Core/MyApp.csproj \
  Core.AutoGenerated/MyApp.AutoGenerated.csproj \
  WebApi/MyApp.WebApi.csproj \
  UnitTest/MyApp.UnitTest.csproj \
  Management/MyApp.Management.csproj
dotnet sln Demo999.sln add \
  Core/Demo999.csproj \
  Core.AutoGenerated/Demo999.AutoGenerated.csproj \
  WebApi/Demo999.WebApi.csproj \
  UnitTest/Demo999.UnitTest.csproj \
  Management/Demo999.Management.csproj

#4. nijo.sln に追加
cd ../..
dotnet sln nijo.sln add "demo/999_NEW_DEMO/Core/Demo999.csproj"                             --solution-folder "デモ999_NEW_DEMO"
dotnet sln nijo.sln add "demo/999_NEW_DEMO/Core.AutoGenerated/Demo999.AutoGenerated.csproj" --solution-folder "デモ999_NEW_DEMO"
dotnet sln nijo.sln add "demo/999_NEW_DEMO/UnitTest/Demo999.UnitTest.csproj"                --solution-folder "デモ999_NEW_DEMO"
dotnet sln nijo.sln add "demo/999_NEW_DEMO/WebApi/Demo999.WebApi.csproj"                    --solution-folder "デモ999_NEW_DEMO"
dotnet sln nijo.sln add "demo/999_NEW_DEMO/Management/Demo999.Management.csproj"            --solution-folder "デモ999_NEW_DEMO"
```

## デモ編集時

### そのデモ独自のコードの場合

デモプロジェクトのソースを直接編集する。

### デバッグコマンドや集約定義に依存しない設定など

デモではなく [テンプレプロジェクト側](../Nijo.NewProjectTemplate/) を編集する。デモで見つかった不具合をプロダクトコードにフィードバックするため。

テンプレプロジェクト側を編集後、 [ソース最新化.bat](../Task/ソース最新化.bat) で最新化する。 VSCode の Run Task からこれを実行する形でも可。

### npmパッケージの新規追加の場合

基本的に各プロジェクトのpackage.jsonではなくモノレポのルートのpackage.jsonに追加すること。
node_modulesの容量削減のため。
