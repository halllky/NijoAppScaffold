# nijo-old からの移植

Nijoは何度か大幅な作り直しが行われている。
旧版（/nijo-old フォルダ）は、Nijoの初期のバージョンであるが、現在のNijo（/Nijo フォルダ）と比べると、かなり異なる構成となっている。
ソースコード自動生成処理の効率性や変更容易性は現在の方が優れているが、実運用に乗っているのは旧版であるため、旧版で使用されている機能を新しいコードベースに移植する必要がある。

* **必須要件**
  * 生成後のソースコードは旧版で生成されるものと一字一句全く同じにする必要がある。
  * 生成に使われるモデル、オプション、コンテキストなどのアーキテクチャやモジュール構成は現在のコードベースにあわせる。
  * スキーマ定義編集GUIは旧版（nijo-old/Runtime/NijoUi.cs）ではなく現在のものに寄せていく。現在の編集GUIの中心となるソースは以下。一部でソースコード自動生成に使われるモデルの種類名などをハードコーディングしている箇所があるため、今回の移植にあわせてそれも対応する必要がある。
    * Nijo/WebService/SchemaEditor
    * Nijo.GuiClient/package_schema-editor-v1
  * 現在のバージョンで定義されている以下の機能は残すこと。旧版からの移植にあたり新旧で同じ物理名が重複する場合、旧バージョンの名称を変えて移植する。（例: ValueObjectModel => ValueObjectModel2）
    * Nijo/SchemaParsing/SchemaParseContext.cs
    * Nijo/SchemaParsing/NodeOption.cs
    * Nijo/Models
    * Nijo/ValueMemberTypes
    * Nijo/Parts
* **任意要件**
  * nijo.xml の書き方、オプションの名前は変更してもよい。
  * カスタム属性など現在のバージョンのみでサポートされている機能は残しても削除してもよい。多くの場合は消すのが大変なので残した方がよいかも。
  * 旧版のソースコードではいろいろなオプションが存在するが、特定の一部のみ使用されているので、それのみ移植すればよい。後述「利用されているもの」を参照。
  * 新バージョンの最小変更にこだわる必要はない。移植前後のソースコードレンダリング処理があまりにも違う場合、無理に同じクラスで新旧両方のレンダリングするのではなく、別クラスを作ってそちらで旧版処理を実装すること。以下、例。
    * 新: DbContextClass.cs
    * 旧: LegacyDbContextClass.cs
* **対応不要**
  * ドキュメントの改訂（/Document フォルダ）
  * デモプロジェクト（/demo フォルダ）


## 進め方

1. [完了] ユニットテストの作成
  * ~~旧バージョンで使われているオプションやデータ構造が定義された nijo.xml を、新旧それぞれのバージョンの記法で定義する。~~
  * ~~移植開始時点ではソースコード自動生成できずエラーになる。移植が完了した時点でエラーが出なくなるはず。~~
  * 作成済み。「旧版と同等内容を新記法で表したスキーマが最終的に旧版と同じソースを生成すること」がこれに該当する。
  * 移行のタスクでは必要に応じて「旧版移植_新記法目標のテスト」にテストを追加してよい。例えば ValueObject のソースだけの新旧比較を行うテストなど。ただし「旧版と同等内容を新記法で表したスキーマが最終的に旧版と同じソースを生成すること」は変更しないようにすること。
2. [完了] ValueObjectModel の移植
  * 旧版の ValueObjectModel と全く同じソースが生成されるように、適宜新バージョンの ValueObjectModel を参考にしながら、 /Nijo/Models フォルダに ValueObjectModel を実装する。
  * 実装中に旧版のオプションが必要になった場合は適宜 /Nijo/SchemaParsing/NodeOption.cs に追加していく。
  * モデル名が新旧で重複しているため、旧版のモデルは ValueObjectModel2 という名前に変えて移植する。
3. [完了] StaticEnumModel の移植 （ValueObjectModel と同様。参考にするのは StaticEnumModel）
4. [完了] WriteModel2 の移植 （ValueObjectModel と同様。参考にするのは DataModel）
5. ReadModel2 の移植 （ValueObjectModel と同様。参考にするのは QueryModel）
  * GenerateDefaultReadModel / GenerateDefaultQueryModel 相当の挙動は ReadModel2 の責務として扱うこと。WriteModel2 側で暫定実装しないこと。
  * ReadModel2 の移植が終わるまでは、WriteModel2 で発生する GDQM 由来の file set 差分（controller や read-model 用 React ファイルの不足・過剰）は未解決のままでよい。WriteModel2 の parity 作業では、GDQM 以外の generated source の差分を優先して潰すこと。
6. GenerateDefaultReadModel / GenerateDefaultQueryModel を移植する。
7. CommandModel の移植 （ValueObjectModel と同様。参考にするのは CommandModel）

その他特筆すべき観点

* 空のスキーマ定義でソース生成した時の出力結果が新旧で同じかどうか。
  * nijo.xml 単位に指定するオプションは存在していてもよいものとする。あくまでデータ構造体やモデルなどルート要素内部のXEelementが無い状態を空とする。
* 「Features」で生成されるソースも移植対象。IFeatureインターフェースは不要。出力後ソースがあっていればよい。
  * 旧版のパス
    * Excel出力: `/nijo-old/Features/Excel`
    * nijo.メッセージ一覧.xml: `/nijo-old/Features/MessageConst`
  * 新バージョンでの移植先
    * `/Nijo/Parts/Common` （C#, TypeScript 両方に出力されるもの）
    * `/Nijo/Parts/CSharp` （C# のみ出力されるもの）
* 旧版互換の作業では、まず file set の差分と content の差分を分けて考えること。
  * file set がずれている場合でも、その原因が未移植の別モデルにあるなら、現在のモデルで責任を持つ generated source の content 差分を先に潰してよい。
  * 特に ReadModel2 未移植の段階では、WriteModel2 のテストに現れる GDQM 由来の webapi / React ページ差分は、WriteModel2 側で場当たり的に埋めないこと。
* 旧版互換の検証は、まず focused test を回し、その結果で最初に壊れている 1 箇所だけを直すこと。
  * WriteModel2 の場合の推奨テスト: `dotnet test Nijo.IntegrationTest/Nijo.IntegrationTest.csproj --filter "WriteModel2が旧版と同じソースを生成すること"`
  * file set 差分で止まっていて、かつ未移植の別モデルが原因だと分かっている場合は、generated source の共通部分（例: `core.AutoGenerated`, `react`）だけを actual / legacy で比較して次の content 差分を探してよい。

#### 注意点: XML上での表現方法

移植前はこのような記述方法を想定している。（nijo-old/Runtime/NijoUi.cs）

```xml
<MyItem is="word max-length:100" />
```

移植後はis属性を廃止し、以下のように指定できるようにする。（Nijo/SchemaParsing/NodeOption.cs）

```xml
<MyItem Type="word" MaxLength="100" />
```

#### 注意点: おいてよい前提

* 新バージョンでは必ず旧バージョン互換モードが指定される前提で実装してよい。 WriteModel2, ReadModel2 などは新バージョンにしか存在しないが、これらを旧バージョン非互換で動かす可能性は排除してよい。
* 旧バージョンでは `UseWijmo` が True の場合に多くのコードが生成対象外になるが、現行システムは True で稼働しており、今後も False になることはないため、このオプションが False の場合のコードは生成対象外にしてよい。

#### 注意点: 複雑な変数マッピングや複雑な構造体の実装方針

複雑な処理、例えば

* ネストされたオブジェクトや配列のレンダリング
* 同じ集約からレンダリングされるとあるオブジェクトから別のオブジェクトへの変換処理
* RefTargetKey, SaveCommand, SearchCondition, EFCoreEntity のように、生成後のクラスやプロパティに複雑な命名規則やネスト規則がある構造体の実装

などについて、旧版では主に `GraphNode<Aggregate>` を中心とした仕組みで複雑なレンダリングを行なっている。
新版では `/Nijo/CodeGenerating/Instance API.cs` で定義されているインターフェースを駆使してこれを解決する。
この種の処理は、その場で文字列を組み立てる補助メソッドを増やして実装してはいけない。Instance API を基準に、生成物に対応するメタデータ構造を先に作ること。
基本的な考え方は以下。

1. レンダリングされるクラス・構造体・フィールド・プロパティ等と 1:1 対応するクラスを定義し、それぞれに以下のインターフェースを実装する。
   * IInstancePropertyOwnerMetadata : フィールドを包含する構造体
   * IInstanceValuePropertyMetadata : フィールド、かつ、フィールドを包含する構造体ではないもの。具体的にはint, string など
   * IInstanceStructurePropertyMetadata : フィールド、かつ、フィールドを包含する構造体
  * 例: エントリークラス、値メンバー、Ref メンバー、Child メンバー、Children メンバーなど。
  * そのメタデータクラス自身が、C# 名、TypeScript 名、プロパティ名、多重度、子メンバー列挙規則を返す責務を持つこと。
2. 処理のレンダリング箇所で `Variable` クラスのインスタンスを宣言し、この変数を起点としてその子孫フィールドを取得する。
   * フィールドの列挙には以下を使う
     * CreateProperties : 構造体直下のフィールドを列挙
     * CreatePropertiesRecursively : ネストされた構造体も含めて全ての子孫フィールドを列挙
     * Create1To1PropertiesRecursively : ネストされた構造体も含めて全ての子孫フィールドを列挙。ただし、元の変数との多重度が 1:1 のものに限る。具体的にはネストされた配列のフィールドとその子孫を除外して列挙する。
   * 何階層もネストされた構造体のレンダリングが基本になるので、多くの場合は再起処理になる。
   * 「配列のときはこういうソースをレンダリングする」「外部参照(RefTo)のときはこう」のように処理を分岐することが多くなるはずだが、これには基本的に型スイッチを用いて上記1で定義したクラスを条件にして分岐をおこなう
   * Variable のルートからそのフィールドまでのパスのレンダリングには基本的に `GetJoinedPathFromInstance` を使う
   * 同じ集約からレンダリングされるとあるオブジェクトから別のオブジェクトへの変換処理において、右辺と左辺でどのフィールドが対応しているかの判定には `ISchemaPathNode.ToMappingKey` を用いる。
3. クラス名やプロパティ名の命名規則は、そのメタデータクラスの中に閉じ込めること。
  * 特定の構造体専用の命名規則を、別の `RenderXXX` メソッドや `GetName` 系の補助メソッド群に分散させないこと。
  * 再起関数の引数でエントリーからのパス名の履歴を持ち回って名前を組み立てるような実装は避けること。
4. オブジェクト変換や対応付けは Instance API で記述すること。
  * `Variable`、`CreateProperties`、`CreatePropertiesRecursively`、`Create1To1PropertiesRecursively`、`GetJoinedPathFromInstance`、`ISchemaPathNode.ToMappingKey` を使い、左辺と右辺の対応関係をコードとして表現する。
  * パス変換が必要な場合は `ISchemaPathNode` の拡張メソッドや、その構造体専用のメタデータクラスを経由して表現すること。
5. 「旧版と同じ文字列が出ること」だけを目的に場当たり的な raw string の分岐や文字列連結で押し切らないこと。
  * まず生成物に対応するメタデータ構造を作り、その上で Render メソッドを書く。
  * 新旧でレンダリング処理が大きく異なる場合は `Legacy*.cs` に分離してよいが、その場合でも命名規則やメンバー構造はメタデータクラスに閉じ込めること。

特に RefTargetKey のような「参照先のキーのみを抜き出した構造」は、単なる集約ツリーの再帰ではなく、生成されるクラス構造そのものを表すメタデータとして設計すること。
`SaveCommand.cs`、`EFCoreEntity.cs`、`SearchCondition.cs` の実装パターンを基準にする。

#### 注意点: 旧版互換モードの責務分離

新旧両対応のコードを書いていると、現行向けの shared source と旧版向けの shared source が互いに上書きし合いやすい。
以下を守ること。

1. legacy 判定は原則として `CodeRenderingContext.IsLegacyCompatibilityMode()` を使うこと。
  * `ForceLegacyCompatibilityMode` のみで分岐すると、旧版互換モデルが含まれる通常の検証時に分岐漏れを起こす。
2. 旧版向けの TypeScript / React ファイルは、現行向けの自動 index 生成や util 生成に上書きさせないこと。
  * 例: legacy の `react/src/__autoGenerated/util/index.ts` は固定内容であり、現行の `IndexTs.Render` の結果で上書きしてはいけない。
3. 集約単位の React ファイルは、旧版では「型定義群」と「関数群」を別々に再配置せず、追加された順序のまま連結される前提である。
  * 旧版互換モードで `SourceFileByAggregate` 相当のファイルをレンダリングする場合、型定義と関数を別配列に分けてから再連結しないこと。
4. 旧版専用の file name, import, util 参照は旧版 branch に閉じ込めること。
  * 例: `batch-update.ts`, `../util2`, `./util` など。
