# はじめに

Nijoは、業務システム（エンタープライズアプリケーション）のスクラッチ開発を効率化するコード自動生成ツールです。
アプリケーションのデータ構造を「スキーマ」として定義し、そこから画面からデータベースまでの定型的なコードを自動生成します。

開発者が書くのは業務固有のロジックだけ。画面、API、型定義、DBのあいだを埋める「つなぎ」の単純作業は、Nijoが引き受けます。

## 解決したい課題

業務システムの開発では、たとえば「商品」というひとつの概念のために、テーブル定義、C#のエンティティ、APIの入出力、TypeScriptの型、画面のフォーム、そして設計書を書きます。
本質的に同じデータ構造を、形を変えて何度も記述し、仕様変更のたびにそのすべてを揃え直すことになります。

複雑なデータ構造と業務ルールの整合性をアプリケーション全体で保ち続けること、そして関係者間で各データの意味や役割の認識を合わせ続けること。これが業務システム開発で困難な部分のひとつです。
Nijoは、この「つなぎ」の作業をなくし、本質的な課題に集中できるようにします。

## 開発の流れ

Nijoでの開発は、おおまかに次の流れで進みます。

### 1. スキーマをGUIで定義する

専用のGUIエディタ（`nijo serve`）で、画面項目、データ構造、参照関係、更新単位を定義します。
データ構造やリレーションはER図のようなグラフとして可視化されるため、「このデータが消えたら、関連するこのデータはどうなる？」「この画面で検索されるデータの構造は？」といったシステムの挙動を、コードを書く前にチームで確認し、実装方針の合意形成を行えます。

この定義がコード生成の唯一の入力になるため、設計書と実装が食い違ったまま放置されることがありません。
編集内容は `nijo.xml` というXMLファイルに保存されます。これが設計ドキュメントであると同時に、コード生成の元データになります。

![GUIエディタ画面](/img/gui-sample-2.png)

### 2. 画面からDBまでのコードが生成される

スキーマから、Entity Framework Core のテーブル定義、C#のクラスとCRUD処理、ASP.NET Core Web API、TypeScriptの型定義とAPIクライアントまでを一気通貫で生成します。
一覧検索や登録・更新・削除といった定型処理も同様です。

フロントエンドからデータベースまで型定義が同期されるため、言語をまたいでデータ構造がずれることはなく、スキーマを変更すれば全体に反映されます。
次の画像は、左がTypeScript、右がC#の生成コードです。

![TypeScriptとC#の構造体が同期される例](/img/lp-002.png)

### 3. 業務固有のロジックだけを手で書く

生成されたコードには、開発者が書き込むための場所が空けてあります。
次の画像では、左が手書きのコード、右が自動生成されたコードです。赤い矢印は、手書きのコードから自動生成された処理を呼び出している箇所です。

手で書く部分（画像左側）:

- ※1: 業務固有のロジック。この例では在庫引当（売上確定時に、どの在庫データをその売上と紐づけるかを決める処理）
- ※2: 更新前の確認メッセージを出すタイミングや文言
- ※3: トランザクションの境界。このほか、UI表現、認証認可、外部システム連携なども手書きの範囲です

自動生成される部分（画像右側）:

- ※4: 楽観排他のバージョンや、登録日時・登録者といった監査用属性の設定
- ※5: 必須、桁数、型など、データ構造から自明に決まる入力チェック

![自動生成コードと手動実装コードの境界を示す画面](/img/lp-001.png)

### 4. UIは案件に合わせて作り込む

自動生成はあくまで下支えです。検索条件や検索結果を画面上でどう見せるかは、通常のTypeScript開発でエンドユーザーや案件に合わせて自由に決められます。

次の画像は、Nijoで構築した販売管理システムの例です。
画面レイアウトは完全スクラッチで、検索ボタンやログアウトボタンを押したときに呼び出すサーバー側処理のデータ構造の部分が自動生成になります。

![Nijoで構築された販売管理システムの画面](/img/lp-003.png)

## モデルによるスキーマ定義

Nijoでは、アプリケーションを「モデル」という単位で定義します。主要なモデルは次の3つです。

- **データモデル**（GUIエディタのオレンジ色の図形）: 永続化されるデータの形。同一トランザクションで更新されるべき複数のテーブルを1つの塊として扱います。
- **クエリモデル**（緑色の図形）: エンドユーザーが検索・一覧表示する粒度のデータの形。
- **コマンドモデル**（青色の図形）: 更新処理、画面初期表示、帳票出力など、入力を受けて結果を返す処理。

この厳格なモデリングにより、データの不整合を防ぎ、保守性の高いシステム設計を促します。
各モデルの詳細は [モデリングの基礎](./02_workflow/020000_modeling.md) を参照してください。

## 技術スタック

Nijoには必須の技術スタックがあります。採用前に注意してください。

- **利用必須**: C# / .NET 9 以上、Entity Framework Core
- **利用推奨**: ASP.NET Core、TypeScript
- **自由**:
  - 任意の JavaScript ライブラリ
  - 任意の .NET ライブラリ
  - EF Core がサポートする任意のRDBMS
