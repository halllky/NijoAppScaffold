# プロジェクト責任者ガイド

案件責任者・技術責任者向けに、Nijoの採用判断と、採用後のプロジェクト管理の観点を整理します。
開発者向けの操作説明ではなく、見積り・体制・品質保証・保守運用の観点からNijoをプロジェクトにどう組み込むかを説明します。

## 採用判断

Nijoは、データ構造とその周辺の処理をモデルとして定義し、そこからソースコードを自動生成するフレームワークです。
開発作業が丸ごと自動化されるわけではないため、採用判断では次の2つを分けて考える必要があります。

* **自動生成で省力化される領域**: データ構造定義、定型的なCRUD、一覧検索（フィルタリング・ページング・ソート）、フロントエンド・バックエンド間の型定義の同期、必須・桁数などの基本的な入力チェック、楽観排他制御
* **引き続き案件側で設計責任を持つ領域**: UIのレイアウトや操作フロー、個別の業務ロジック、外部システム連携、トランザクション範囲の設計、運用設計

この切り分けを受け入れられる案件ほど、Nijoとの相性が良くなります。

### 向いている案件

* データ構造の変更や追加が多く、フロントエンド・バックエンド・データベースの整合性維持が負担になりやすい
* 一覧検索、登録更新、参照といった定型処理が多い
* 開発メンバーの増減が多く、新規参画メンバーに設計の全体像を共有し続けたい
* 業務データの完全性（排他制御や同時更新によりデータが壊れないこと）が重要である
* 管理対象項目の増減や変更が将来にわたって発生する見込みが高く、初期の学習コストよりも中長期の変更容易性を重視する

### 慎重に判断すべき案件

* 画面表現やUXの独自性が主な競争力であり、データ処理の定型部分が少ない
* 既存システムの資産や既存設計書を強く前提としており、モデル中心の整理に切り替えにくい
* 複数の開発会社で同じコードベースを共有しており、かつ他社の開発を制御できないなど、チーム全体でスキーマ定義を中心に設計を進める運用に乗れない
* 短期間の小規模改修で、導入学習コストを回収しづらい

なお、小規模案件やPoCでも、将来の本開発を見据えてモデリング方針を先に固めたい場合には採用する意義があります。
その場合は最初から全面採用を目指すよりも、データ構造の整理と主要な定型処理の自動生成に絞って評価するほうが現実的です。

## 開発工程ごとの活用方法

V字モデルの工程ごとに、Nijoが何を担い、どの作業が従来のスクラッチ開発と同様に残るかを整理します。
Nijoは工程をなくすのではなく、工程ごとの作業内容と重心を変えます。

### 要求整理〜外部設計

外部工程は基本的にNijoの射程外です。
外部工程では「動かなくてもよいからイメージを形にする」モックアップが重視されますが、Nijoが生成するコードは実運用を想定したものであり（未実装部分は厳密にエラーになる）、UI自体も自動生成されないためです。

ただし、次のような部分的な活用は可能です。

* GUIエディタを、依頼者や開発者間の認識合わせの道具として使う
* 実際にデータ構造や画面項目を定義してみて、設計に矛盾がないかを確かめる

また、もし次のようなワークフローを確立できれば外部工程の効率化も可能です。

* 自動生成されない部分をAIエージェントに実装させる仕組みを構築し、高速なモックアップ作成を実現する

### 内部設計〜実装

Nijoの恩恵を最も受けられる工程です。
何が自動生成され、何が手動実装かは「採用判断」に記載の通りです。
開発の進め方は [設計・開発](./02_workflow/index.md) を、生成されるコードの構造は [アーキテクチャ概要](./04_architecture.md) を参照してください。

管理上の注意点は2つあります。

1つ目は、**ソースコードの自動生成は一度かけて終わりではなく、開発期間を通じて継続的に行われる**ことです。
スキーマ定義の変更とコードの再生成は、日常の開発サイクルの一部になります。

2つ目は、**スキーマ定義も実装の一部**としてそれなりのウェイトを占めることです。
誰がスキーマ定義を行い、誰がそのレビューを行うかを早い段階で決めてください。
ここが曖昧なままだと、生成コードに何を期待してよいかがメンバーごとにずれやすく、導入効果が下がります。

役割分担の一例:

* 案件責任者: Nijoで省力化する範囲と、案件側で個別設計する範囲を決める
* 技術責任者: モデル設計方針、生成コードの扱い、カスタマイズ方針を決める
* 開発者: `nijo.xml` の更新、生成コードの利用、個別実装を行う
* レビュー担当者: モデル定義、生成コード利用方法、手実装ロジックの境界を確認する

### テストと品質保証

自動生成された部分は、毎回ゼロから仕様確認する対象ではなく、既知の生成規則に基づく部分として扱えます。
不具合は多くの場合、手動実装の部分に集中します。
したがって管理者にとって重要なのは、「何をテストしなくてよいか」よりも「どこにテスト資源を集中させるか」を明確にすることです。

品質保証計画では、次のように分けて考えるのが現実的です。

* **Nijoの標準構成を前提に確認する項目**: スキーマ変更が型定義、DBスキーマ、定型CRUDに正しく反映されること
* **案件側で重点的に検証する項目**: UIの業務適合性、権限制御、外部連携、運用監視、性能、セキュリティ統制
* **両者をまたいで確認する項目**: ログ設計、マイグレーション手順、障害時の復旧手順、変更時のレビューとリリース運用

最終的にどこまでテストするかは、案件ごとの品質方針で決めてください。

また、スキーマ定義では機能間の関連性が明示されるため、どの機能を動かすとどのデータが更新され、後続のどの画面に影響するのかを把握しやすくなります。
これはテスト計画の立案や、テスターと設計者の間の仕様認識合わせにも役立ちます。

非機能要件は機能テストとは別に整理が必要です。
[非機能要件への対応方針](#非機能要件への対応方針) の通り、Nijoが標準でカバーするのはデータ完全性、更新時の整合性、保守しやすい構造などであり、認証認可、可用性、監視運用、災害対策、性能保証は案件側の基盤設計や個別実装と組み合わせて担保する必要があります。

### 保守運用

保守運用フェーズではNijoを以下のように活用できます。

#### オンボーディング

開発者と運用担当者が別のチームになる現場では、運用担当者がシステムの構造を理解するための情報共有が重要になります。
Nijoのスキーマ定義は、通常のドキュメントと異なり実装と同期されている保証があるため、運用担当者はこれを信用してシステムの理解を始められます。

#### 影響調査・変更管理

モデルの変更は、関連するコードに広く影響します。
ただし、生成コードには強い型定義が付くため、影響範囲は再生成時のコンパイルエラーとして表面化します。

たとえば以下の画像では、左側のペインでデータ項目名が「仕入単価_税抜」から「仕入単価_税込み」に変更されています。
この状態でソースコードを生成しなおすと、右側や下側のペインのように、変更前の名前を前提としていた実装箇所がコンパイルエラーとして現れます。

![モデル変更後に旧項目名参照がコンパイルエラーとして表面化する例](/img/compile-error.png)

変更の影響範囲を人手で全面的に洗い出さなくても、型に現れる破綻は再生成とビルドで機械的に炙り出せます。次の手順を変更管理プロセスに組み込んでください。

1. モデルを変更する
2. コードを生成しなおす
3. ビルドして、旧定義に依存していた実装をコンパイルエラーとして洗い出す
4. 洗い出された箇所を修正し、レビュー対象を確定する

変更管理の単位は「ソースファイル単位」ではなく「モデル変更と再生成のセット」として扱うことを推奨します。
これにより、仕様変更時のレビュー観点を揃えやすくなります。

#### 破棄容易性

Nijoが生成するのは、モダンな技術スタックで構成された「普通の」ソースコードです。
ブラックボックスなランタイムライブラリへの依存は最小限で、システムの詳細はすべてソースコードとして出力されるため、運用担当者がコードを直接確認して障害調査を行えます。

この方針は、**いつでもNijoを捨てられる**ことも意味します。

* 生成コードが原因のバグが発生した場合、緊急対応として該当箇所を手修正して対応できる
* 運用が長引きNijoの知識を持つ担当者がいなくなった場合、自動生成をやめて以降はすべて手修正で運用すると決められる（ただしこれは最終手段であり、基本的にはNijoを使い続けるほうが開発生産性の恩恵を受けられます）

## 非機能要件への対応方針

Nijo の標準構成が IPA「非機能要求グレード」の各観点をどこまでカバーするかを整理します。
ここでいう標準構成とは、Nijo が生成する CRUD 処理・一覧検索処理・ASP.NET Core Web API・TypeScript 型定義・EF Core データアクセス・アプリケーションサービス層の拡張ポイント、およびマイグレーション／ログ／設定外出しの土台を指します。

### 全体像

Nijo はデータ完全性・検索機能・保守性の土台を揃えることに強みがあります。
一方、高可用構成・認証認可・監視運用・災害対策といったインフラ寄りの非機能要件は、導入先の基盤設計や個別実装と組み合わせて実現する前提です。

非機能要件は、標準構成に対して次の3段階で整理できます。

* **土台がある**: データ完全性、更新時の整合性、検索機能の標準化、保守しやすいコード構造、設定外出し、マイグレーション
* **一部対応（チューニング前提）**: 性能チューニングの余地、運用標準化、内部統制向けの証跡基盤、外部システム連携の構造化
* **対象外（個別設計が必須）**: 認証認可、通信保護、高可用構成、監視運用、災害対策、移行計画、インフラ設備条件

Nijo は「非機能要件をすべて内包した完成済み業務パッケージ」ではなく、「業務アプリケーションの堅い土台を生成するフレームワーク」です。

### Nijo が提供する土台

観点別に、標準構成が寄与する主な点を挙げます。
ここに挙げていない項目（冗長化、監視、死活監視、DR、認証基盤、性能保証、インフラ設備など）は基本的に対象外であり、導入先の基盤設計やアプリ個別実装に委ねられます。

#### 可用性・データ保全

* 更新系はトランザクションとロールバック前提で生成され、障害時にデータ不整合を残しにくい。更新・削除・論理削除ではセーブポイントも利用される。
* 楽観排他制御とユニーク制約により、更新競合や更新失敗時のデータ破損を防ぐ。
* ただしこれは「止まりにくい構成」ではなく「壊れにくい更新処理」に近い。冗長化や災害対策は別途必要である。

#### 性能・拡張性

* 一覧検索は Skip / Take によるページングとソートを型付きで標準提供し、全件取得を避けやすい。
* QueryModel は DB View へのマッピングに対応し、複雑な参照系を RDBMS 側で最適化できる（View 定義自体は手動）。
* 一般的な ASP.NET Core / EF Core 構成のため、スケールアップやコンテナ配備へ移行しやすい（自動水平分散は未提供）。

#### 運用・保守性

* 自動生成コードと手動実装コードが分離され、再生成時に手修正が潰れにくい。業務ロジックはアプリケーションサービス層に集約される。
* 更新者・更新時刻などの監査列を持つ標準構成があり、内部統制向けの証跡の土台になる。

#### 移行性

* マイグレーション・設定外出し・リリース時のバージョン記録により、継続的な改修や再配備はしやすい。
* 既存資産からのデータ変換や業務切替は業務知識を伴うため、自動生成の対象外である。

#### セキュリティ

* DataModel に集約単位の楽観排他制御が組み込まれ、更新競合を検知できる。スキーマからユニーク制約やバリデーションを定義できる。
* ASP.NET Core と型付き API により安全な実装を行いやすい（CSRF・CSP・Cookie 方針・入力サニタイズは個別確認が必要）。
* 認証・認可は標準では固定されない。現在ユーザーの取得方法も未定義扱いのため、認証基盤との接続は個別実装が必要である。デモ101 のログイン機構は標準機能ではなく実装例である。

### 権限設計の指針

認可は個別実装ですが、Nijo では次の方針を推奨します。

* **機能単位の権限**
  * カスタム属性での実装を推奨。Query / Command モデルごとに権限の種類を設定する（「システム管理者のみ利用可能」など）。
  * ASP.NET Core の ActionFilter で属性を参照し、認可処理を実装する。
* **データ単位の権限**
  * 閲覧（QueryModel）: `CreateQuerySource` でログインユーザーの権限を見て Where 句を付加する。
  * 閲覧・更新（CommandModel）: コマンドハンドラー内で対象データへの権限を判定する。
  * DataModel: どの CommandModel からでも常に権限判定が必要な場合は、更新前チェックとして実装することも可能である。

### 導入判断の目安

次のような案件に特に適しています。

* 業務データの整合性を重視する基幹寄りの業務アプリ
* 一覧検索や CRUD の品質を早期に揃えたい案件
* 将来の保守や引継ぎを考慮し、素直な C# / TypeScript コードを残したい案件

逆に、厳格な認証認可やセキュリティ統制、24時間365日運用や高い SLA、大規模・高負荷での性能保証、複雑な既存システムからの移行が主眼である場合は、Nijo 単体ではなく周辺基盤や個別設計を前提に採用判断すべきです。
