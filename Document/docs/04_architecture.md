# アーキテクチャ概要

このページでは、自動生成コードと手動実装コードがどのように分離・統合されるかと、
Nijo を用いたアプリケーションのアーキテクチャ構成を説明します。

アーキテクチャ構成では Web アプリケーションの場合とそれ以外の場合を分けて説明します。

## 自動生成と手動実装の分離

自動生成されるコードは `__AutoGenerated` などといった名称のフォルダの中に生成されます。これらと開発者が手動で実装するコードとは、明確にファイルやフォルダが分離されています。
スキーマ定義を変更してコードを再生成しても、手動実装されたビジネスロジックやUIコンポーネントが上書きされて消えることはありません。

以下は Nijo を用いた典型的なプロジェクトのソースコード構成例です。オレンジ色の部分がソースコード生成のたびに毎回洗い替えられます。手動実装コードはオレンジ色の部分の外に置くことで、再生成しても消えないようになっています。

![自動生成コードと手動実装コードの分離イメージ図](/img/workflow.drawio.svg)

## 画面をもつ Web アプリ

Nijoが生成するアプリケーションは、基本的にはモダンなWebアプリケーション構成（SPA + REST API）をとります。

### レイヤー構成

標準テンプレートは、一般的な3層アーキテクチャを採用しています。

![Nijo の標準アーキテクチャ図](/img/architecture-overview.drawio.svg)

黄色が自動生成される領域、赤色が開発者が手動で実装する領域です。レイヤーごとに責務を分けつつ、再生成しても手動実装側が上書きされない構成になっています。

各モデルは、次のように各層へ配置されます。

- **Command / Query Model**: フロントエンド（APIクライアントと型定義）からバックエンド（Controller とビジネスロジック本体）までをまたいで配置されます。
- **Data Model**: ビジネスロジック層からデータアクセス層を通じてデータベースに永続化されます。フロントエンドには直接露出せず、必ず Command / Query を介して操作されます。

#### 1. プレゼンテーション層（Web / UI）

- **役割**: ユーザーインターフェースの提供とリクエストの処理。
- **フロントエンド**: APIクライアントと型定義（TypeScript）が自動生成されます。UIライブラリやCSSフレームワークの選定・画面構築は完全に自由で、採用必須なのはTypeScriptのみです。
- **バックエンド（Controller）**: モデルごとのコントローラーとアクションが自動生成されます。役割はビジネスロジック層を薄くラップする程度です。認証・ロギングなどWebサーバーとしての挙動は、通常のASP.NET Coreと同様に手動で実装します（`.AddControllers()` で自動生成コントローラーも統合されます）。

#### 2. ビジネスロジック層（Core）

- **役割**: 業務ルールの実装、データの加工、トランザクション管理。Command / Query Model の処理本体や Data Model がここに置かれます。
- **Application Service**: プレゼンテーション層からの要求を受け、データの登録・更新・削除・検索やその他複雑な処理を実行します。ライフサイクルはリクエスト1回分（Webであれば1リクエスト、バッチであれば1実行）であり、状態を持ってはいけません。
- **自動生成と手動実装の分離**: 「ジェネレーションギャップパターン」を採用しています。中心となるサービスクラスは `abstract class` として自動生成され、開発者はこれを継承した具象クラスでメソッドをオーバーライドして振る舞いを拡張します。

#### 3. データアクセス層（Infrastructure）

- **役割**: Entity Framework Core を介したデータベースとのやり取り。
- **自動生成**: スキーマ定義に基づく `DbContext`、Fluent API によるモデル定義処理 `OnModelCreating`、エンティティクラス。
- **手動実装**: 自動生成される `OnModelCreating` のあとに、任意の `OnModelCreating` 処理を追加できます。

### 登録更新画面のランタイム遷移

レイヤー図だけでは、実際にどのオブジェクトがリクエストごとに生成され、どこで値が詰め替えられるかが見えにくいため、登録更新画面の実行時の流れも図にします。以下はデモ101の「売上詳細」画面を題材に、画面初期表示から更新確定後の画面再読込までを追ったものです。

この図で重要なのは、初期表示も保存も CommandModel のエンドポイントとして扱われる点と、サーバー側では `IPresentationContext` に ReturnValue や Messages を詰めて返す点です。クライアント側はその戻り値として `DisplayData` を受け取り、保存成功後は `navigate` または `revalidate` により同じ初期表示フローをもう一度通して最新状態を読み直します。

## 画面をもつ Web アプリ以外

前項で説明した、画面ありWeb以外のアプリケーションも構築できます。

### HTTPで呼び出す（Web API 連携、AIエージェント）

QueryModel, CommandModel の Web エンドポイントはUIに依存しない入出力方式となっています。
**外部システム** や **AIエージェント** に対してこれらのエンドポイントを公開することで、UIなしの Web API アプリケーションとして運用することもできます。

### C#メソッドで呼び出す（デスクトップアプリ、コマンドラインアプリ）

Nijoが生成するビジネスロジック層（`MyApp.Core`）は、特定のUIフレームワークに依存しない純粋な .NET クラスライブラリです。
デモでは ASP.NET Core と React を採用していますが、これら2つのソースを利用せず、 `MyApp.Core` を直接参照し、アプリケーションサービスのメソッドを呼び出すことで、**Windows Forms** や **WPF**、**コンソールアプリケーション(コマンドラインアプリケーション)** など、全く異なるプラットフォームのアプリケーションを構築することも可能です。
