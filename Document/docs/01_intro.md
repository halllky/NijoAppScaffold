# はじめに

**複雑な業務を、目に見える形に。スキーマ駆動アプリケーション生成フレームワーク**

Nijoは、複雑な業務フローを持つエンタープライズアプリケーションを効率的に開発するための、フルスタック・コード生成フレームワークです。
データ構造と振る舞いを定義したスキーマを中心とし、堅牢なアプリケーションの土台を自動生成します。

## 特徴

業務アプリケーション開発において最も困難なのは、複雑なデータ構造と業務ルールの整合性を保ち続けることです。
Nijoは、 **「モデル (Model)」** という概念を用いてそのアプリケーションのスキーマを定義し、専用のGUIエディタで可視化することで、この課題を解決します。

### 1. 可視化による設計の共有
専用のGUIエディタ (`nijo serve`) を提供しています。
複雑なデータ構造やリレーションをER図のようなグラフで可視化することで、エンジニア間での仕様確認や認識合わせをスムーズに行えます。
「このデータが消えたら、関連するこのデータはどうなる？」「この画面で検索されるデータの構造は？」といったシステムの挙動を、コードを書く前に視覚的に共有し、実装方針の合意形成を行えます。

### 2. スキーマ駆動のモデリング
Nijoでは、「モデル」という単位でデータを捉えます。
モデルは、その種類に応じて、どのようなソースコードが自動生成されるかが変わります。
代表的なモデルの種類は以下3つです。

- **データモデル**: RDBMSのテーブル定義と近いですが、単なるテーブル定義ではなく、同一トランザクションで更新されるべき複数のテーブルを1つの塊と捉えます。データモデルからは Entity Framework Core の定義や、必須チェックや桁数チェックなど基本的な入力検証などのコードが生成されます。
- **クエリモデル**: エンドユーザーが検索・一覧表示する粒度でモデルを定義します。クエリモデルからは、検索条件オブジェクト、検索結果オブジェクト、動的WHERE句付加、ORDERBY句、ページングなどのコードが自動生成されます。
-  **コマンドモデル**: 処理。「xxxデータ更新」「ログイン」「xxxデータ複製」「月次xxx帳票作成」など、エンドユーザーやスケジューラが起点となり、Webクライアントとサーバーが協働して実現する必要のあるものがこれにあたります。

この厳格なモデリングにより、データの不整合を防ぎ、保守性の高いシステム設計を強制します。

### 3. フルスタックかつ型安全なコード生成
定義されたモデルから、以下のレイヤーのコードを一気通貫で生成します。

- **データアクセス層**: Entity Framework Core によるテーブル定義
- **ビジネスロジック層**: モデルごとのC#クラス定義, CRUD処理
- **プレゼンテーション層**: TypeScript 型定義, ASP.NET Core Web API, それと対応するJavaScript側のAPIクライアント

フロントエンドからバックエンド、データベースまで型定義が同期されるため、変更に強い開発が可能です。

## V字モデルにおける位置づけ

Nijoを用いた開発は、一般的なウォーターフォール開発（V字モデル）の各工程を以下のように変革します。
Nijoのスキーマ定義は、主に**開発チーム内部での設計図**として機能します。

### 内部設計 （開発チーム内部でのデータ構造定義、機能分割）
Nijoのスキーマ定義 (`nijo.xml`) の作成は、この工程に相当します。
会社やチームによって「基本設計」「詳細設計」といった言葉の定義は異なりますが、ここでは開発チーム内部のコミュニケーションや認識合わせのために用いられる設計を指して内部設計と呼んでいます。

Nijoのスキーマ定義には、例えば以下のような、エンドユーザーには関心のない属性・実装上の都合による属性が含まれます。そのため、エンドユーザーとの合意形成を目的とした外部設計書というよりも、開発チーム内部での認識合わせや設計共有を目的としたドキュメントとして位置づけられます。

* ユーザーには見えない内部用のUUIDによるキー
* システム管理者や保守運用担当者のためのフラグ
* パフォーマンス最適化のためのキャッシュ用テーブル

ただし、もしエンドユーザー側の発注者がシステム設計に明るかったり、こういった内部設計に理解がある場合、Nijoのスキーマ定義を「機能一覧」や「アプリケーションのスコープ定義書」として認識合わせに利用することが可能かもしれません。

### 設計ドキュメントとしての観点からのスキーマ定義
nijo.xml をドキュメントとして見たとき、どういった情報が信頼可能かを説明します。

* ✅️ **必須チェックや各項目最大桁数といった、Nijoによってカバーされる仕様** については、nijo.xmlの内容を正として進めることで、常に実装と同期された信頼可能なドキュメントとみなすことが可能です。ただし、正式な納品物に転用したい場合、nijo.xmlをExcelやHTMLなどのヒューマンリーダブルな形式に変換する工夫が必要かもしれません。
* ⚠️ **処理のロジックや振る舞い** について、必須チェックなどといった挙動については上記の通りです。一覧検索処理についてはクエリモデルが大部分をカバーしているため相当な省略が可能です。「ログイン」「xxxデータ更新処理」「xxx帳票出力処理」といったコマンドモデルが担当する処理については、nijo.xmlに記載できるのは、それらの入出力の項目定義と、処理概要を自然言語で記述するのが精一杯です。そこで表現しきれない複雑な仕様については、別途管理が必要です。
* ❌️ 認証・認可やロギングといった共通機能や、UIコンポーネントの挙動など、 **ソースコード自動生成されない要素** については、このツール以外での管理が別途必要です。

### 実装
ボイラープレートコードの記述が不要になり、開発者は「自動生成ではカバーできない固有のロジック」の実装に集中できます。

### 単体テスト
自動生成されたコード（基本的なCRUD処理やバリデーション）については、Nijoフレームワークによって品質が担保されるため、単体テストを省略できます。
開発者は、自らが手動で実装したビジネスロジックやUIの挙動のテストに注力してください。

## 技術スタック

Nijoが生成するアプリケーションは、以下の技術スタックを採用しています。

- **データベース**: 任意のRDBMS (EF Coreがサポートするもの) / デフォルトテンプレートはSQLite
- **バックエンド**: .NET 9 / ASP.NET Core / Entity Framework Core
- **フロントエンド**: TypeScriptのみ採用必須。それ以外のJavaScript, CSS フレームワークやライブラリの採用は任意。
